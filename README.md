# Windows 10 SSH Brute Force Detection & Investigation

## Objective

This project simulated a real-world SSH brute force attack against a Windows 10 endpoint to validate the organization's ability to detect unauthorized authentication attempts, investigate suspicious login activity, and identify the source of the attack using centralized log analysis.

The objective was to generate realistic attack telemetry, collect endpoint logs through Sysmon and Windows Event Logging, forward security events into Splunk SIEM, and perform a complete investigation from detection to attribution.

---

## Scenario

A Windows 10 workstation hosting an OpenSSH service was targeted by an attacker attempting to gain unauthorized access through repeated password guessing attacks.

Using a Kali Linux attack machine, multiple SSH authentication attempts were launched against the target endpoint. The activity generated numerous failed login events, creating a recognizable brute force pattern within the Windows Security logs.

The SOC analyst was tasked with investigating the authentication anomalies, identifying the source of the attack, and determining whether any successful compromise occurred.

## Attack Simulation

The attack was conducted from a Kali Linux system against the Windows 10 endpoint running the OpenSSH service.

A password wordlist was used to generate multiple authentication attempts against the local user account.

```bash
hydra -t 10 -V -f -l The-Analyst -P passwordlist.txt ssh://192.168.10.10
```

This activity produced a high volume of failed authentication events that were subsequently collected and forwarded to Splunk for analysis.

---

## Investigation Process

### Step 1 – Initial Detection

The investigation began by reviewing Windows Security Event Logs ingested into Splunk.

A noticeable spike in failed authentication attempts was observed against a single user account over a short time period, indicating potential brute force activity.

---

### Step 2 – Event Analysis

Relevant Windows Security Events were examined to identify:

* Target username
* Source IP address
* Authentication status
* Event frequency
* Attack timeframe

The analysis revealed repeated login failures originating from a single external source.

---

### Step 3 – Source Attribution

By pivoting on the originating IP address, the attacker system was identified as:

**Source IP:** `192.168.10.30`

The source corresponded to the Kali Linux attack machine operating within the lab environment.

---

### Step 4 – Timeline Reconstruction

Using Splunk search capabilities, a timeline of authentication activity was constructed.

The investigation showed:

* Multiple failed login attempts
* Consistent targeting of the same account
* Rapid execution pattern typical of automated password attacks
* No evidence of lateral movement or post-exploitation activity

---

## Detection Logic

A custom Splunk search was developed to identify excessive authentication failures within a defined time window.

The detection focused on:

* Failed SSH logins
* Authentication frequency thresholds
* Source IP correlation
* User account targeting patterns

This approach helps security teams rapidly identify brute force attacks while reducing false positives generated by normal user behavior.
